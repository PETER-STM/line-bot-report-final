【行李袋 C:AI 協作流程 — 最終定稿】
(三個Claude推導 + 2026業界對撞驗證 + 元原則閉環)

═══ 一、核心認知(最根本,先讀)═══
Claude Code = 有手的腦,不是純執行器。
- 它跟對話框的claude是同一個Claude,能規劃討論反駁,還多一雙手(看得到碼、能跑)。
- 舊習慣(Gemini遺留):對話框規劃→搬去Claude Code執行 = 在「看不到碼的地方」規劃,多搬運且品質低。
- 正解:日常規劃+寫+改直接在Claude Code一站完成(Plan Mode想清楚再動手)。
- 對話框的claude留給:戰略討論、當Breaker挑盲點、卡住換腦袋——價值正是「沒看到你的碼」的乾淨外部視角。

═══ 二、跳不跳畫面:95/5原則 ═══
- 日常95%(改bug/加功能/改邏輯):一站在Claude Code完成,不跳畫面。
- 高風險5%(碰錢/碰DB/改認證/重建核心引擎):才跳一次,貼來對話框當Breaker攻一次→判斷→回去改。
- 不是「反覆橫跳」,是「刻意花一次跳的成本,買獨立視角守高風險」。
- 判準:改錯會不會釀大禍、難不難回收?會+難→值得跳一次。不會+好回收→別跳。
- (比舊習慣的「每件事都先來對話框」大幅減少跳的次數)

═══ 三、七步SOP(每次改碼都用)═══
動手前→①能小改不大改 ②只讀必要範圍(叫AI只讀第X-Y行) ③指令精準到不能誤解
執行時→④先給diff不直接寫 ⑤一次改一處
驗證→⑥機器驗證(py_compile)不信「你說改好了」;失敗時把錯誤訊息「原文整段貼回」,不轉述、不自己診斷〔親驗:今日貼回修對〕 ⑦同處失敗兩次就換方法

═══ 四、Claude Code內建升級(回電腦逐步試,一次學一個)═══
- Plan Mode(Shift+Tab兩次進、一次出):先規劃給你看、批准才動手。就是「動手前先想清楚」的原生位置。
- Hooks(Stop hook):改完自動跑py_compile,沒過不准結束。
- 模型分級:Sonnet做一般,卡住才切Opus。
時機:先用熟的方式把部署修好上線,之後改碼才試Plan Mode。別死線前學新工具。

═══ 五、雙AI工作法(高風險改動才用)═══
角色:Builder(提案/寫碼) + Breaker(專職攻擊) + 你(唯一拍板)
循環:Builder寫→你過眼→丟Breaker攻→你判攻擊真假→成立的丟回Builder修→機器驗證收尾
Breaker開場指令:「我貼你另一個AI的方案/碼,你唯一任務是找它哪裡會爆。不附和不客套,攻擊要具體(X情況下會Y)。攻不動就直說,別硬擠毛病。」
鐵則:①你每步在迴路裡,不准兩AI直接對話 ②機器驗證是最終裁判,非AI共識 ③攻擊收斂條件:一輪攻不出「新類別」破綻就停(邊際遞減即停),不是無上限攻到天荒地老
何時用:核心邏輯/碰錢碰DB/架構決策。改錯字加log別用,殺雞不用牛刀。

═══ 六、最高原則:分清「工具強制」vs「人腦判斷」═══
- 能被Hook/檔案焊死的(失敗攔截、py_compile關卡、日誌讀寫)= 真機械強制,高壓不破功
- 註定靠人腦的(攻擊深度該不該停、方案對不對)= 老實承認,別假裝機械化
- 危險:把人腦判斷貼上「工具強制」標籤→以為焊死了而放心→高壓時破功
- 第二層判斷(能焊≠該焊):焊死本身有成本。落地前先比「誤殺成本 vs 漏接成本」哪邊痛。Hook是全域攔截,觸發條件寫鬆會擋掉正常結束,反而跟自己閘門打架。能焊的不一定都該焊,別一把梭。

═══ 七、三點流程強化(回電腦碰Hook/repo才實作)═══
1. 失敗攔截機械化:同一錯誤第二次,用Stop hook「真的攔截」(不是寫進prompt假裝強制,那高壓下照樣破)。「同處」要定義為「同根因」,不是同行號(否則AI把錯誤位移到別行就繞過閘門)。
2. 攻擊深度:碰錢/DB/認證才攻深,但用「邊際遞減即停」收斂,不開無上限(無上限=你變唯一終止鈕,判斷成本變無界,打臉省額度原則)。
3. 失敗歸檔+召回:在「放棄某方法時」就記(不只成功才記);記的是「決策軌跡+在哪步歪」不只記結論;存進repo的檔(Claude Code開工會讀);要有「下次試類似方法時自動把舊紀錄推到眼前」的召回機制——否則是沒人掃的墳。記錄不難,到點召回才是關鍵。

═══ 八、核心心法(數字僅為方向性佐證,源頭未親驗)═══
- 省額度本質=少重做(訂閱制,省的是額度內做更多)
- 你的真瓶頸是「審查頻寬」非寫碼速度【論據:你一人六日,審查速度固定,AI寫再快也卡在你這關——結構事實】。改動要小到你審得懂,否則大diff囫圇放行=技術債默默堆積。〔未驗證佐證:有報告稱無治理AI生成PR審查等待較長,源頭未親驗〕
- 第⑥條原文貼回:今日親身驗證有效(報錯貼回,claude修對)。〔親驗 n=1;另有業界文章稱此法修正率高,源頭未親驗〕
- 問題多在「流程」不在「模型」【論據:你今日六次失敗皆流程問題,非模型能力】。〔親驗 n=1;另有評測稱頂尖模型真實任務分數遠低於benchmark,具體數字未親驗〕
- 業界方向是更嚴格驗證非全自動;碰錢永遠人工把關;你是判斷節點
- 注意:上述親驗條目目前 n=1,是「當前最好的工作假設」,非結案定律。用第七節補丁3的失敗日誌持續餵養,讓 n 從1往上長。把心法當待證假設庫,比當定稿更穩。

═══ 九、元原則:防「冒充可信」機制(今日親身驗證,連claude都會犯)═══
規律:白皮書/AI/工具/甚至你自己,都會不自覺給「未驗證的主張」穿上可信外衣(華麗詞/數字/權威來源),讓它跳過你的查證。善意者也會犯,且越信任的來源外衣越厚、越危險。

防法(機械閘,不靠意志力):
①承重才查:凡要拿來撐決策的主張,不分來源強制查一手來源;不承重的標「未驗證」即可。判準是「承不承重」,不是「信不信這來源」。
②證據分級標籤,固化進文件:每個關鍵主張標〔親驗〕/〔一手已查〕/〔未驗證佐證〕。不禁止用未驗證的東西,只禁止它「冒充已驗證」。剝掉外衣,機器就空轉。
③對最信任的來源(含claude、含你自己)反而設更高查證——它們外衣最厚,最易讓你跳過驗證。

閉環(防使用端越權):
- 標籤擋得住「冒充已驗證」(來源端偽裝),但擋不住「未驗證的東西日後被升格使用」(使用端越權)。主張的承重程度會隨時間漂移:今天判它不承重、沒查,三個月後被人直接拿去推論,它就事後變承重了,而當初沒查源頭。
- 補法:〔未驗證佐證〕標籤後一律加觸發句「若要拿此條撐決策,先補查」。把「承重判定」從一次性事前動作,改成綁在使用時點的閘——跟補丁3「到點召回」同一思路:記錄不難,到引用點觸發查證才是關鍵。

來源誠實三條(防「把取得方式講得比實際高」):
第一條·來源等級必須明示,用詞分級鎖死。任何承重的外部事實宣稱,入場即帶取得方式標籤,用詞不可越級:
  〔親驗〕= 我親眼跑過/你親身驗證過(最高)
  〔一手已查〕= 讀過「該事實的原始產生者」(官方文件/論文原文/原報告/法規本身),網址指向那個原始出處——不是指向任何轉述它的二手頁面。能給出一個打得開的網址,不自動等於一手。
  〔搜尋所得·來源未明〕= 搜尋引擎/AI生成/二手文章讀來,未追到原始產生者
  〔未驗證〕= 沒查
  鐵律:沒有指向原始產生者的網址,不准用「一手」「查證」「已驗證」。「我搜到了」只能標〔搜尋所得·來源未明〕,絕不升級成〔一手已查〕。
第二條·禁止用「動作詞」冒充「等級」。查了/看了/搜了/確認了=動作,不是等級。每個動作後面必須接它實際到達的等級標籤。違規樣態:用「已查證」「一手圖像是」「報告說」卻給不出指向原始產生者的出處。
第三條·降格/修正後的新結論,同樣受前兩條管。最隱蔽的破口:糾正一個錯宣稱時順手帶進的「新事實」(替代名單/機制/數字),會因包在「我已查證糾正了」的語氣裡而免標來源。鐵律:糾正不是免標金牌,降格後端出的任何新外部事實,一律重新套第一條標等級。

剎車:別翻到「全不信」的另一極端(同樣癱瘓)。不承重的佐證,允許它以「未驗證」身份留著當參考。分寸=降格,不刪除。三條只管「承重的外部事實」;閒聊/口語/比喻/你的業務經驗(那是親驗)/純邏輯推理/常識,不適用——否則來源誠實會從「防外衣的閘」變成「拖慢一切的儀式」,另一種過度工程。

═══ 十、利用率診斷(Claude表現不如預期時,依序問)═══
① 是天花板還是利用率?(它做不到 vs 我沒餵好)
② 若是利用率,我在哪個模式?——工程模式通常已開滿;漏水幾乎都在探索模式(prompt鬆、指涉混淆)。
③ 修補手段是調自己的輸入濃度,不是加規則去管模型。
(註:此條本身即在說「別靠堆規則」。C到此節為止,夠了——回去執行,別再加。)
