## 失靈案例登記(可入失靈日誌)

### 失靈 #1:中文內容過聊天管道吃字(同型多次,最嚴重)
**型別:中文重打損壞**
- 現象:讓 Claude Code 把 CLAUDE.md v4 全文「重打」寫入,經「貼進對話框→CC 讀進變數」管道時被截斷/吃字。多處條款缺字(「不入法」「止回報」整段消失),且脚本本身語法殘缺(`with open()` 缺冒號、`.readlin` 截斷)。
- 發生次數:文字重打吃字 ×2、脚本 OLD 轉義被還原成中文 ×1、脚本 `-c` 套娃截斷 ×1、`.readlin` 顯示截斷疑慮 ×1 = 同型 5 次
- 為何危險:顯示出來的內容 ≠ 磁碟真身,肉眼 diff 看不出缺字,差點 commit 殘缺檔
- **已有守則命中**:v4「禁止直接執行長 heredoc」「以磁碟為準、ascii 抽查」「複雜脚本先落檔自檢」——守則對,但 CC 仍反覆嘗試違規寫法,需要人(Lyon)每次手動攔

### 失靈 #2:git 狀態誤判導致六層彎路
**型別:git status 誤讀**
- 現象:最早 `git status` 顯示「untracked:無」,被解讀成「定稿.md 不存在/沒追蹤」,於是一路查 .gitignore→子目錄→core.excludesFile→改檔名→force add,六層全錯
- 真相:該檔**早已 commit 進 repo(8441971)**,「untracked 無」是因為「已追蹤且無改動」,本來就不該出現在 untracked 列表
- 代價:六層彎路 + 一次非必要的中文→英文改名(雖然改名本身有獨立價值,但當下是基於錯誤前提做的)
- 根因:沒在第一時間用 `git ls-files` 確認追蹤狀態,而是從 `git status` 反推

## 預防措施(下次怎麼不再踩)

### 對失靈 #1(中文吃字)——升級為硬規則
1. **大段中文內容進 repo,一律用「檔案傳輸」不用「聊天管道重打」**:
   - 由對話框 Claude 生成 .md 檔 → 下載 → 檔案總管覆蓋進 repo → CC 只負責「讀磁碟驗證」
   - 絕不讓 CC 把貼進去的中文全文「重打」成新檔(那條管道會吃字)
2. **跨端 md5 驗證納入標準流程**:源頭(生成端)算 md5 → 落地端用 verify.py 比對。PASS 才算完整。比 ascii 抽查更徹底。
3. **含中文的小範圍改動(數行),直接手動改(VS Code),不寫脚本**:今天手改一秒解決脚本試四次沒成的事。v4 已有此條,要更早採用,不要等脚本失敗四次才想到。

### 對失靈 #2(git 誤判)——加一條認知
4. **判斷檔案在不在 repo,用 `git ls-files`,不從 `git status` 反推**:
   - 「git status untracked 無」≠「檔案不存在/沒追蹤」,可能是「已追蹤且無改動」
   - 動手解「git 看不到檔案」這類問題前,先 `git ls-files | grep 檔名` 確認真實追蹤狀態,再決定方向

### 對「兔子洞」本身——流程級
5. **失敗兩次換方法,要真的執行**:今天脚本落盤失敗到第四次才換手動。「同處失敗兩次」的計數要由人主動喊停,CC 不會自己停(它會一直推測新原因)。
6. **疲勞尾段不開精細活**:這個子任務(改一行指針)在已經很長的當下開工,違反 v4 疲勞警戒。低價值 + 高耗神 = 該延後或用最笨方法(手動)。
